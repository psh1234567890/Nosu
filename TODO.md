# 노수베스트 자동화 TODO 큐

이 파일은 자동화 에이전트가 다음 작업을 고르는 단일 작업 큐다.

## 큐 사용 규칙

- 자동화는 아래 `## 대기` 섹션에서 가장 위에 있는 `[대기]` 티켓 1개만 처리한다.
- 작업 시작 전에는 해당 티켓에 적힌 `직접 관련 파일`만 우선 읽는다. 기본 2~3개를 넘기지 않는다.
- 작업 범위가 티켓 밖으로 커지면 구현하지 말고 TODO에 후속 티켓을 추가한다.

## 대기

### [완료] T01. 운영 정책 설정 패널을 실제 서비스용으로 분리

- 목표: 토론 시간, 보상 코인, 신고/제재 기본값처럼 서비스 운영자가 조정해야 하는 값을 코드 상수에만 두지 않고 관리자 화면에서 확인/저장할 수 있게 한다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 서버 상태에 `platformSettings` 또는 동등한 설정 객체를 추가한다.
  - 기본값은 현재 코드 상수와 동일하게 유지한다.
  - 관리자 전용 API로 설정 조회/저장을 제공한다.
  - 저장 시 허용 범위 제한을 넣는다. 예: 발언 시간 30~600초, 보상 코인 0~10000, 신고 임계값 1~20.
  - 관리자 UI 운영 탭에 설정 요약과 편집 폼을 추가한다.
  - 설정 변경은 audit log에 남긴다.
- 완료 조건:
  - 기존 데모 상태에서도 설정 기본값이 자동 보정된다.
  - 잘못된 설정값은 400 응답으로 거절된다.
  - 저장 후 새로고침해도 설정이 유지된다.
- 제외:
  - 실제 관리자 권한 체계 재설계는 하지 않는다.
  - 전체 UI 리디자인은 하지 않는다.

### [완료] T02. 회원가입/로그인 이후 필수 약관 동의 흐름 추가

- 목표: 공개 서비스에 필요한 이용약관, 개인정보 처리방침, 커뮤니티 규칙 동의를 계정 상태에 저장하고 미동의 사용자는 핵심 기능 진입 전에 동의를 받는다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 사용자 모델에 약관 동의 버전, 동의 시각, 동의 IP 또는 최소 메타데이터를 추가한다.
  - 신규 가입자는 기본적으로 최신 약관에 동의한 상태로 생성하거나, 첫 로그인 직후 동의 화면을 거치게 한다.
  - 기존 데모/기존 사용자에는 안전한 마이그레이션 기본값을 적용한다.
  - 약관 버전이 올라가면 기존 사용자에게 재동의를 요구할 수 있는 구조를 둔다.
  - 관리자/데모 로그인 흐름이 깨지지 않게 한다.
- 완료 조건:
  - 미동의 사용자는 아레나, 지갑, 프로필 편집 등 핵심 행동 전에 동의 UI를 본다.
  - 동의 저장 후 원래 화면으로 정상 진입한다.
  - 동의 상태가 `/api/auth/session` 또는 상태 payload에서 안전하게 확인된다.
- 제외:
  - 실제 법무 문안 작성은 placeholder 문구로 둔다.
  - 외부 전자서명 서비스 연동은 하지 않는다.

### [완료] T03. 프로필 인증 클레임 심사 큐 구현

- 목표: 사용자가 학력/경력/전문성 클레임을 제출하고, 운영자가 승인/반려/보류할 수 있는 최소 심사 흐름을 만든다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 클레임에 제출 사유, 증빙 텍스트/URL, 제출 시각, 심사자, 심사 메모를 저장한다.
  - 사용자 프로필에서 새 클레임 제출 폼을 제공한다.
  - 관리자 화면에서 pending 클레임 목록, 승인, 반려, 메모 저장을 제공한다.
  - 승인/반려 시 사용자 알림과 audit log를 생성한다.
  - 반려된 클레임은 공개 프로필에서 과도하게 노출하지 않는다.
- 완료 조건:
  - 일반 사용자가 클레임을 제출하면 관리자 큐에 보인다.
  - 관리자가 승인하면 프로필 신뢰도/인증 배지에 반영된다.
  - 반려하면 사용자 알림이 생성되고 상태가 `rejected`로 남는다.
- 제외:
  - 파일 업로드 스토리지는 별도 티켓에서 처리한다.

### [완료] T04. 신고 처리 큐를 운영 가능한 상태 머신으로 확장

- 목표: 신고가 단순 목록에 머물지 않고 접수, 검토중, 조치 완료, 기각 같은 상태로 흘러가며 운영자가 처리 이력을 남길 수 있게 한다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 신고 상태를 `open`, `reviewing`, `resolved`, `dismissed`로 확장하거나 동등 구조를 추가한다.
  - 관리자 화면에서 상태 변경, 담당자, 처리 메모를 저장한다.
  - 신고 대상이 채널/메시지/사용자인 경우 요약 링크 또는 식별자를 보여준다.
  - 상태 변경은 audit log와 신고자/대상자 알림에 반영한다.
- 완료 조건:
  - 데모 신고를 생성/처리할 수 있다.
  - 처리 완료된 신고는 기본 목록에서 접히거나 필터링된다.
  - 신고 export에 처리 메타데이터가 포함된다.
- 제외:
  - 자동 제재 판단 AI는 만들지 않는다.

### [완료] T05. 제재 만료/해제와 기능 제한을 실제 행동에 연결

- 목표: 경고/정지 제재가 단순 기록에 그치지 않고 로그인 후 행동 제한, 만료 표시, 운영자 해제까지 연결되게 한다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 활성 정지 사용자는 채널 생성, 토론 참여, 메시지 작성, 투표 등 핵심 행동이 막히게 한다.
  - 제재 만료 시간이 지나면 자동으로 비활성 처리되거나 응답에서 비활성으로 계산한다.
  - 관리자 화면에서 제재 해제와 해제 사유를 남길 수 있게 한다.
  - 사용자에게 현재 제재 상태와 남은 시간을 보여준다.
- 완료 조건:
  - 정지 사용자는 API와 UI 양쪽에서 핵심 행동이 차단된다.
  - 해제된 제재는 audit log에 남고 즉시 행동 가능해진다.
  - 만료된 제재는 활성 제재 카운트에서 빠진다.
- 제외:
  - 외부 본인확인/신원 제재 DB 연동은 하지 않는다.

### [완료] T06. 알림 센터를 읽음/미읽음/액션 링크 중심으로 정리

- 목표: 토론 결과, 신고 처리, 제재, 클레임 심사 같은 이벤트를 사용자가 놓치지 않도록 알림함을 실제 제품 수준으로 다듬는다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 알림 목록에 미읽음 필터, 전체 읽음, 개별 읽음 처리를 추가한다.
  - 알림에 `view`, `roomId`, `channelId`가 있으면 클릭 시 해당 화면으로 이동한다.
  - 오래된 알림을 접거나 최근 N개 중심으로 보여준다.
  - 관리자/사용자 역할별 알림 문구가 어색하지 않게 정리한다.
- 완료 조건:
  - 새 알림 발생 후 미읽음 배지가 증가한다.
  - 클릭 시 관련 아레나/관리자/프로필/지갑 화면으로 이동한다.
  - 읽음 처리 후 배지가 줄어든다.
- 제외:
  - 푸시 알림, 이메일 알림은 하지 않는다.

### [완료] T07. 비공개 채널 초대 코드 관리 기능 추가

- 목표: 비공개 토론방의 초대 코드가 한 번 생성되면 끝나는 구조가 아니라, 운영자/방장이 재생성하거나 비활성화할 수 있게 한다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 채널 생성자 또는 운영자가 초대 코드 재생성/비활성화를 할 수 있게 한다.
  - 비활성화된 코드로 입장하면 명확한 오류를 보여준다.
  - 초대 코드 변경은 audit log 또는 채널 이벤트로 남긴다.
- 완료 조건:
  - 기존 코드로 입장 가능하다.
  - 재생성 후 이전 코드는 실패하고 새 코드는 성공한다.
  - 비활성화된 비공개 채널은 직접 코드 입장이 막힌다.
- 제외:
  - 친구 목록/DM 초대 UI는 만들지 않는다.

### [완료] T08. 토론방 탐색 UX를 검색/정렬/상태 필터 중심으로 개선

- 목표: 방과 채널이 늘어나도 사용자가 원하는 공개 토론을 빠르게 찾을 수 있게 한다.
- 직접 관련 파일: `src/App.tsx`, `src/styles.css`
- 구현 범위:
  - 아레나에 검색어 입력, 상태 필터, 정렬 옵션을 추가한다.
  - 정렬 후보: 최신순, 참여 대기 우선, 라이브 우선, 종료 토론 우선.
  - 필터 결과가 없을 때 빈 상태와 재설정 버튼을 보여준다.
  - 모바일과 중간 폭에서 검색/필터가 겹치지 않게 한다.
- 완료 조건:
  - 데모 데이터에서 검색어와 상태 필터 조합이 정상 동작한다.
  - 딥링크로 들어와도 선택된 채널이 필터 때문에 사라질 경우 안내가 보인다.
- 제외:
  - 서버 사이드 페이지네이션은 하지 않는다.

### [완료] T09. 음성 토론 기기/재연결 상태 패널 강화

- 목표: 실제 사용자가 마이크 권한, 연결 실패, 상대방 이탈 상황을 이해하고 복구할 수 있게 한다.
- 직접 관련 파일: `src/App.tsx`, `src/styles.css`
- 구현 범위:
  - 마이크 권한 실패, peer 연결 실패, 재연결 중, 연결됨 상태를 더 명확히 표시한다.
  - 음소거/손들기 상태와 실제 연결 상태를 분리해서 보여준다.
  - 상대방이 나갔을 때 대기/재초대/텍스트 전환 안내를 제공한다.
- 완료 조건:
  - 정상 연결 시 `connected` 상태가 사용자에게 명확히 보인다.
  - 권한 실패 또는 연결 실패 시 다음 행동이 보인다.
- 제외:
  - SFU/미디어 서버 도입은 하지 않는다.

### [완료] T10. AI 판정 결과 이의제기/재판정 요청 흐름 추가

- 목표: AI 판정이 끝난 뒤 사용자가 이유를 확인하고, 운영자에게 재검토를 요청할 수 있는 최소한의 공정성 장치를 만든다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 종료된 토론 결과에 이의제기 버튼과 사유 입력을 추가한다.
  - 서버에 이의제기 기록을 저장하고 중복 요청을 제한한다.
  - 관리자 화면에서 이의제기 목록과 처리 상태를 볼 수 있게 한다.
  - 처리 완료 시 요청자에게 알림을 보낸다.
- 완료 조건:
  - 종료 토론에서 1회 이의제기 제출이 가능하다.
  - 같은 사용자의 중복 제출은 제한된다.
  - 관리자 처리 후 상태와 알림이 갱신된다.
- 제외:
  - 실제 AI 재호출/승패 자동 변경은 별도 티켓으로 둔다.

### [완료] T11. 지갑 원장 상세와 운영자 조정 기능 구현

- 목표: 코인 지급/차감/구매/보상 이력을 사용자가 이해하고, 운영자가 안전하게 수동 조정할 수 있게 한다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 지갑 화면에 원장 상세, 유형 필터, 메모 검색을 추가한다.
  - 관리자 전용 코인 지급/차감 폼을 추가한다.
  - 수동 조정에는 사유를 필수로 받고 audit log에 남긴다.
  - 음수 잔액 방지 제한을 서버에 둔다.
- 완료 조건:
  - 사용자는 본인 원장을 필터링해서 볼 수 있다.
  - 운영자는 특정 사용자에게 코인을 지급/차감할 수 있다.
  - 음수 잔액이 되는 차감은 거절된다.
- 제외:
  - 실제 결제/환전/현금성 재화 처리는 하지 않는다.

### [완료] T12. 계정 데이터 내보내기와 개인정보 삭제 요청 초안 구현

- 목표: 사용자가 본인 데이터 다운로드와 삭제 요청 상태를 볼 수 있는 1차 개인정보 대응 흐름을 만든다.
- 직접 관련 파일: `server/index.js`, `src/App.tsx`
- 구현 범위:
  - 사용자 본인 데이터 export API를 만든다.
  - 프로필/계정 화면에 내 데이터 다운로드 버튼을 추가한다.
  - 계정 삭제 요청은 즉시 물리 삭제가 아니라 요청 상태를 남기고 관리자 큐로 보낸다.
  - 관리자 화면에 삭제 요청 목록과 처리 메모를 추가한다.
- 완료 조건:
  - 본인 데이터 export에는 다른 사용자의 secret이 포함되지 않는다.
  - 삭제 요청 후 사용자는 요청 상태를 확인할 수 있다.
  - 관리자 처리 기록이 audit log에 남는다.
- 제외:
  - 법적 보존 기간 계산 자동화는 하지 않는다.

### [완료] T13. Supabase normalized 모드 실서비스 마이그레이션 점검 강화
